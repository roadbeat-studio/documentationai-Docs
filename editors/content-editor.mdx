## Overview

The content editor is where you spend most of your time in Studio. It renders a **dynamic form** based on the content type's schema, organizing fields into tabs and sections for a clean editing experience.

## Editor Layout

The editor consists of several areas:

- **Toolbar** (top) — Save, Publish/Unpublish buttons, and status indicators
- **Tab bar** — Navigate between content sections (e.g., Basic Info, Content, Media, SEO)
- **Form area** (center) — The actual fields you fill in
- **Status bar** (bottom) — Current status, version number, last saved time

## Working with Tabs

Content types with a **form schema** organize fields into tabs. Common tab layouts include:

| Tab | Typical Fields |
|-----|---------------|
| **Basic Info** | Title, slug, description, category |
| **Content** | Article body, structured text, main content |
| **Media** | Hero image, gallery, video attachments |
| **SEO** | Meta title, meta description, keywords, OG image, canonical URL |
| **Settings** | Publication date, access type, language |

Click a tab to switch between sections. Fields in the current tab are validated inline as you type.

## Basic vs Extended Mode

Many content types have fields marked as **extended** — advanced fields that most editors don't need daily. Toggle between modes using the **Basic / Extended** switch in the toolbar.

- **Basic mode** — shows only essential fields for a streamlined editing experience
- **Extended mode** — shows all fields including advanced metadata and configuration

<Callout kind="tip">
  Your mode preference is saved per content type in localStorage, so it persists across sessions.
</Callout>

## Common Form Section Expansion

Form schemas can reference reusable **common form sections** via `$ref` entries in `layout.tabs` or `layout.sections`. When the content editor loads a form schema, the `SchemaRefResolverService` automatically expands these references:

1. Loads the referenced common section file (e.g. `seo-section.schema.json`)
2. Applies `exclude_fields` to remove unwanted fields
3. Applies `field_overrides` to customize labels, help text, or validation
4. Merges the resolved fields into the form's `fields` array

This happens transparently — editors see the expanded fields inline without needing to know about the `$ref` mechanism. The canonical **SEO section** is included via `$ref` in all 50 content types, providing consistent meta title, meta description, and OG image fields.

<Callout kind="info">
  The slug field is **not** part of the SEO section. For content types with a primary title field, the slug is auto-generated and displayed inline below the title. For content types without a title field (e.g. `quote`, `check-in`, `joke`), an explicit `identifier.slug` field is defined in the content type's form schema and rendered via the same slug editor component.
</Callout>

<Callout kind="info">
  `$ref` expansion works in both tab-based and section-based form layouts. The resolver processes `layout.tabs` and `layout.sections` identically.
</Callout>

## Auto-Save

The editor automatically saves your work:

- **Debounce interval**: 5 seconds after the last change
- **Status indicator**: Shows "Saving...", "Saved", or "Unsaved changes" in the status bar
- **Manual save**: Press **Ctrl+S** / **Cmd+S** to save immediately

## Unsaved Changes Protection

If you try to navigate away with unsaved changes, a confirmation dialog appears asking if you want to:

- **Save and leave** — saves changes, then navigates
- **Leave without saving** — discards unsaved changes
- **Stay** — cancels navigation

This is implemented via Angular's `CanDeactivate` guard.

## Version History

Every save creates a new version. Access version history from the status bar:

<Steps>
  <Step title="Click the version number" icon="history">
    In the status bar (e.g., "Version 5"), click to open the version history panel.
  </Step>
  <Step title="Browse versions" icon="list">
    Each version shows who made the change, when, and a summary of what changed.
  </Step>
  <Step title="Restore if needed" icon="rotate-ccw">
    Select a previous version and click **Restore** to create a new version with that data. The current version is preserved in history.
  </Step>
</Steps>

<Callout kind="info">
  CE retains up to 10 versions per content item. Pro has unlimited version history.
</Callout>

## Publishing from the Editor

The top-right toolbar shows context-sensitive actions:

| Content Status | Available Actions |
|---------------|-------------------|
| **Draft** | Save, Publish |
| **Published** | Save (creates new version), Unpublish |
| **Archived** | (Read-only) |

When you click **Publish**:

1. All required fields are validated
2. The content status changes to `published`
3. The Delivery API starts serving the content
4. If Discovery Nodes are configured, a signed teaser is distributed

## Validation

Fields are validated both inline (as you type) and on save/publish:

- **Required fields** — highlighted with a red border if empty
- **Min/max length** — character count shown for text fields
- **Pattern validation** — regex patterns for emails, URLs, etc.
- **Cross-field date validation** — an `after_field` rule on a date field ensures it cannot be set before the referenced field (e.g. end date must be after start date); the field border turns red and an inline error message appears
- **Custom rules** — content-type-specific validation

Validation errors are shown inline below each field and summarized in a notification if publish is blocked.

## Nested Field Paths

Form schemas use **dot-notation field IDs** to reference nested properties in the content data structure. For example, a field with `field_id: "editorial.is_breaking"` maps to the nested path `data.editorial.is_breaking`.

The content editor handles this transparently:

- **Reading** — `getFieldValue("editorial.is_breaking")` traverses the nested object path `data → editorial → is_breaking`
- **Writing** — `updateField("editorial.is_breaking", true)` creates intermediate objects as needed and sets the value at the correct nested path
- **All field types** — dot-notation is supported for all accessor methods: `getStringValue()`, `getBooleanValue()`, `getNumberValue()`, `getArrayValue()`, `getLocalizedStringValue()`, and `getBlocksValue()`

<Callout kind="info">
  Top-level field IDs (without dots) continue to work as before — they read and write directly on the root data object.
</Callout>

## Asset Picker Modal

When you click an image, gallery, or file field placeholder (or the **Replace** button on an existing asset), the **Asset Picker Modal** opens as a full-screen overlay:

- **Search** — type in the search bar to filter assets by filename
- **Folders** — browse the folder tree in the left sidebar
- **Upload** — click the **Upload** button or drag-and-drop files directly onto the modal
- **MIME filtering** — the picker automatically filters by media type based on the field (e.g. only images for an image field, only videos for a video gallery)
- **Single select** — click an asset to select it immediately (for single image/media/file fields)
- **Multi select** — check multiple assets, then click **Select N Assets** (for gallery fields)
- **Adaptive page size** — the number of thumbnails per page adjusts automatically to fill the available space
- **Pagination** — Prev/Next buttons are pinned below the grid, always visible without scrolling

<Callout kind="tip">
  Uploading a file in single-select mode automatically selects the newly uploaded asset and closes the picker.
</Callout>

## Structured Subforms

Complex object and array fields render as **structured subforms** instead of raw JSON textareas. The form schema's `ui.component` value for a field determines which subform pattern is used. There are two patterns:

### Object Subforms

A single object field renders as an inline bordered card with labelled inputs for each sub-field. No add/remove controls — the object always exists and you edit its properties directly.

### Repeater Subforms

An array field renders as a **repeater** — a vertical stack of cards, each representing one item, with a dashed **Add** button at the bottom. Each card has:

- A **header bar** showing the item number and a brief summary of its key fields, plus a **Remove** button
- **Structured inputs** for all sub-fields of the item

<Callout kind="tip">
  The card header summarises the most important fields so you can scan a long list at a glance without expanding each card.
</Callout>

## Location Search

Content types with location fields (coordinates, addresses) get an integrated **location search** powered by OpenStreetMap. The search component appears at the top of any form section that contains a coordinates field, above the individual address fields.

<Steps>
  <Step title="Search for a location" icon="search">
    Start typing a place name, address, or landmark in the **Search Location** field. After 3 characters, matching results appear in a dropdown powered by the OpenStreetMap Nominatim geocoding API.
  </Step>
  <Step title="Select from results" icon="map-pin">
    Click a result (or use arrow keys + Enter) to select it. A **banner-style map** appears showing the pinned location on an interactive OpenStreetMap tile layer.
  </Step>
  <Step title="Address fields auto-fill" icon="check-circle">
    When you select a location, all related address fields in the same section are automatically populated:

    - **Street Address** — road and house number
    - **City** — city, town, or village
    - **Region / State** — state or county
    - **Postal Code** — postcode
    - **Country** — if a **Country Select** dropdown exists in the same section, the correct country is automatically selected based on the ISO 3166-1 alpha-2 code returned by OpenStreetMap
    - **Coordinates** — latitude and longitude
  </Step>
</Steps>

The search component always appears **above** the individual address fields to match the natural address-entry flow: search first, then review and fine-tune. The latitude and longitude fields remain manually editable below the map. The search supports two common schema patterns — nested address fields (`location.address.street`) and flat fields (`location.street_address`).

<Callout kind="tip">
  The location search uses the free OpenStreetMap Nominatim API — no API key or external service configuration is required.
</Callout>

## Video Picker

The **video picker** component (`ui.component: "video_picker"`) provides two entry modes for video fields:

<Tabs>
  <Tab title="Asset Library" icon="film">
    Click **Select video from asset library** to open the Asset Picker modal filtered to `video/*` MIME types. After selecting a video, a native `<video>` player renders inline with playback controls.

    The footer shows a green **From asset library** badge and offers **Replace from library** and **Remove** actions.
  </Tab>
  <Tab title="External URL" icon="link">
    Paste a YouTube, Vimeo, or direct video URL into the **External video URL** input.

    - **YouTube** — all common URL formats are auto-detected (`youtube.com/watch?v=`, `youtu.be/`, `/embed/`, `/shorts/`, `youtube-nocookie.com`). Renders as an embedded YouTube player (16:9 iframe).
    - **Vimeo** — `vimeo.com/<id>` URLs are detected. Renders as an embedded Vimeo player.
    - **Direct URLs** — `.mp4`, `.webm`, `.ogg`, `.mov` URLs render via a native `<video>` element.
    - **Other URLs** — displayed as a clickable link card.

    The footer shows a blue badge indicating the provider (**YouTube embed**, **Vimeo embed**, or **External URL**). The URL can be edited inline without removing and re-adding the video.
  </Tab>
</Tabs>

<Callout kind="info">
  YouTube and Vimeo iframe URLs are sanitized via Angular's `DomSanitizer` and cached per URL to prevent unnecessary re-sanitization on change detection cycles.
</Callout>

## Context Directory Proxies

The Studio API proxies two commonly used Context Directory endpoints so that the editor frontend does not need direct access to the Context Directory service:

| Studio Endpoint | Proxied From | Cache | Used By |
|----------------|-------------|-------|---------|
| `GET /api/v1/currencies` | Context Directory `/api/v1/currencies` | 1 hour | Currency Select dropdown |
| `GET /api/v1/countries` | Context Directory `/api/v1/countries` | 1 hour | Country Select dropdown, Location Search auto-fill |

Both endpoints include a built-in fallback list (major currencies / EU + US countries) that is returned if the Context Directory is unreachable, so the editor remains functional even without the external service.

## Layout Preview

The editor includes a **live layout preview** panel on the right side (on wide screens) that renders your content using the content type's layout definition. The preview updates in real time as you edit fields.

- **Image fields** — resolved from asset objects to display the actual image
- **Image galleries** — rendered as a 3-column thumbnail grid
- **Video galleries** — rendered as stacked video players with filename labels
- **Date/datetime values** — ISO strings (e.g. `2026-03-31T19:00`) are formatted as readable dates (`Mar 31, 2026, 7:00 PM`)
- **Enum labels** — i18n labels are displayed instead of raw enum codes
- **Breakpoint toggle** — switch between desktop, tablet, and mobile previews
