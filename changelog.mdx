## Content Reader, Dynamic Pod Features & Tip System

**Inline Content Reader**
- **Three display modes** — 4K ultrawide (≥2560px) side-by-side split layout, regular desktop/tablet/mobile full-screen modal overlay, and external content new-tab opening with link indicator badges
- **Content Reader component** — renders full pod content with hero image, article body with rewritten asset URLs, publisher metadata, reading time, word count, categories, tags, gallery, and content signature verification badge
- **Content Reader modal** — ARIA dialog with focus trap, Escape to close, backdrop click to close, slide-up animation (respects `prefers-reduced-motion`), and focus restoration on close
- **Content Reader placeholder** — shown in the 4K split layout right pane when no content is selected, with keyboard navigation hints (↑↓ navigate, Enter select, Esc close)
- **External link badge** — overlay badge on all 5 card variants (grid, compact, feed, large list, large tile) indicating content that opens in a new tab

**Dynamic Pod Features**
- **Pod reactions** — emoji reaction bar (👍❤️🔥👏🤔) with optimistic toggle updates, count display, active state styling, and support for pod-specified custom reaction types
- **Pod comments** — threaded comment display with recursive rendering up to configurable `maxDepth`, comment submission form with name/email/content fields, reply-to threading, loading skeletons, and empty state
- **Pod newsletter** — inline subscription form with email input, loading spinner, success confirmation, error handling, and publisher name personalization
- **Pod-proxy BFF** — all dynamic pod interactions proxied through the Studio BFF via `POST /discover/pod-proxy`, targeting `/_dynamic/api/reactions.php`, `/_dynamic/api/comments.php`, and `/_dynamic/api/newsletter.php`
- **Capability-driven rendering** — dynamic sections only appear when the pod's `_roadbeat/config.json` advertises the feature as enabled

**Tip System**
- **Tip data model** — `TipRecord`, `TipBudgetConfig`, and `TipStats` types with localStorage persistence for both tip history (last 500 entries) and budget configuration
- **Budget tracking** — monthly budget (default 100 units), computed spending stats (this month spent, remaining, top 5 publishers), and `canTip` guard that prevents exceeding the budget
- **Tip multiplier UI** — 1x–5x pill selector in the content reader footer with budget progress bar (green/yellow/red), "Tipped!" confirmation badge, and disabled state when budget is exhausted
- **Auto-tip** — configurable countdown timer (default 30s) that starts when content loads, displays a live countdown in the footer, and automatically fires a 1x tip when it reaches zero; respects budget limits and fires only once per content item
- **Tip budget settings** — new "Tip Budget" section in Discover Settings with monthly budget slider (10–500), auto-tip toggle, delay slider (10–120s), auto-tip multiplier pills (1x–5x), spending stats summary, and clear old history action

**Studio as Content Source**
- **Local content serving** — Content Reader fetches published content directly from the Studio database when available, eliminating the need for a deployed Content Pod in development, demos, and early-stage publisher setups
- **`CANONICAL_URL_BASE` configuration** — new environment variable controls the base URL for published teasers; set to `http://localhost:3000` in dev so `classifyContentSource()` routes content to the local database instead of treating it as external
- **Automatic source detection** — `getContentForReader()` checks the local database first; if published content exists for the requested content type and slug, it serves it directly without querying Discovery Node teaser endpoints
- **Content transformation** — `transformStudioContentToPod()` maps Prisma database records to the `PodContentItemDto` format expected by the Content Reader, including headline, description, article body, media, classification, dates, publisher info, and metadata

**Map & Globe Interactions**
- **OpenStreetMap popup click handlers** — clicking the hero image, title, or teaser description in a map popup opens the Content Reader (modal or 4K split pane); "Why am I seeing this?" and Bookmark buttons are fully wired with `stopPropagation`
- **3D Globe popup click handlers** — hero image and title clicks open the Content Reader via `ngZone.run()` to ensure Angular change detection fires from outside the zone; Why This and Bookmark buttons work with `stopPropagation`

**Polish**
- **Dark mode audit** — verified all new components have comprehensive `dark:` Tailwind variants; fixed external link badge bindings on feed card variant
- **Enhanced loading skeleton** — realistic full-layout skeleton matching content reader structure: hero image, type badges, title, metadata row, article paragraphs, reaction pills, comment placeholders, and footer skeleton
- **Context-specific error states** — 4 distinct error screens: pod unreachable (orange shield), content not found (gray document), signature verification failed (red warning triangle), and generic error — each with descriptive copy, error detail, Retry button, and Close link
- **4K split-pane reactivity fix** — converted `@Input() data` to a signal-backed setter so all `computed()` properties (title, body, hero, etc.) re-evaluate when switching between cards in the side-by-side layout
- **Content body readability** — article text constrained to `max-w-2xl` (672px) and centered for comfortable reading on wide screens
- **Responsive hero image** — `h-[55vh]` with `max-h-[600px]` and `min-h-[280px]`, vertically centered via `object-center`; ensures headline and article start are always visible below the image
- **Compact footer** — Bookmark, Share, and Original links placed on the left; Tip slider on the right in a single row on wide screens, with reduced padding for a slimmer bar
</Update>

## Discover UI refinements, boolean filter fix & content editor nested paths

**Discover UI**
- **Humanized content type names** — a new `HumanizeContentTypePipe` transforms content type slugs (e.g., `blog-post`) into display names (`Blog Post`) across all view modes: grid cards, compact cards, feed cards, large list, large tile, content type badges, bookmark popovers, and the transparency drawer
- **Filter header styling** — content-type-specific filter headers now use a violet background matching the active filter badge color, with no borders and equal top/bottom padding
- **i18n active filter labels** — active filter badges now resolve i18n-translated labels from facet bucket data instead of displaying raw enum codes (e.g., "Analysis" instead of `analysis`)
- **Boolean toggle switches** — boolean facets (e.g., "Exclusive", "Breaking News", "Free content") now render as compact one-liner toggle switches instead of checkboxes, with aggregation doc counts displayed next to the label

**Boolean filter fix**
- **Root cause** — boolean filters returned no results because the form editor saved dot-notation field IDs (e.g., `editorial.is_breaking`) as flat object keys instead of nested object paths, causing the `enrichTypeFilters` schema walker to read stale seed data instead of user-edited values
- **Form editor fix** — `updateField()` and all field accessors (`getFieldValue`, `getBooleanValue`, `getStringValue`, etc.) now handle dot-notation field IDs as nested object paths via new `setNestedValue()`/`getNestedValue()` helpers
- **`walkSchemaForFilters` fallback** — the teaser generator's schema walker now accepts a `rootData` parameter and falls back to flat dot-notation keys in the root data when the nested value is missing, ensuring compatibility with legacy content data
- **Boolean pass-through** — reverted boolean-to-string normalization (`true` → `"1"`) since Elasticsearch already has boolean-typed `type_filters` fields; booleans are now passed through as-is, with the Discovery Node query builder coercing `"1"`/`"0"` to `true`/`false` on the query side

**Active filter badge labels**
- **Content type humanization** — active filter badges humanize content type values (e.g., "News" instead of `news`)
</Update>

## Schema-driven content-type-specific filters in Discover

**Teaser enrichment**
- **`enrichTypeFilters()`** — generic schema-driven method in `TeaserGeneratorService` that reads `x-filter` annotations from the content type's metadata schema (via Schema Registry) and extracts filterable values into `teaser.type_filters` at publish time
- **Pattern support** — handles all x-filter extraction patterns: simple values (A), object with value_path (B), arrays (C), arrays of objects with value_path (D)
- **Non-fatal** — if the Schema Registry is unreachable, the teaser is still valid without `type_filters`

**Consumer BFF**
- **`translateBffFilters()` rewrite** — cleanly passes through `type_filters.*` keys to the Discovery Node alongside existing DN-native paths
- **`FacetGroupDto` expanded** — added `content_type_specific`, `ui_hint`, `unit`, `labels`, `priority` fields to the BFF facet DTO
- **Facet metadata pass-through** — both BFF facet mapping locations now forward type-specific metadata from DN aggregation responses

**Discover UI**
- **Computed signals** — `selectedSingleContentType`, `globalFacets`, `typeSpecificFacets` in `DiscoverStateService` for reactive type-specific filter rendering
- **`ContentTypeFilterHeaderComponent`** — colored divider between global and type-specific filters with content type icon, accent color, and active filter count
- **Type-specific filter section** — appears in the facet sidebar when a single content type is selected, showing schema-driven filters below the global ones
- **Chips widget** — new pill/chip toggle rendering for small enum facets (≤6 values or `ui_hint: "chips"`)
- **Range units** — range filter inputs now display unit labels (m², min, €) from the schema
- **Auto-clear** — type-specific filters (`type_filters.*`) are automatically cleared when the content type selection changes to multi or none
- **Violet filter pills** — active type-specific filters show as violet-tinted dismissible pills in the `ActiveFiltersComponent`, visually distinct from global brand-colored pills
</Update>

## Calendar week view & dynamic teaser count

**Week view drill-down**
- **"+N more" switches to week view** — clicking the "+N more" link on any day in the monthly calendar now opens a dedicated week view (Monday–Sunday) for that day's week, showing all teasers without truncation
- **Full teaser list per day** — week view displays every teaser with title, color-coded content type badge, and publisher name; each day column has a scrollable area filling the available viewport height
- **Week navigation** — previous/next week arrows in the header; date range label (e.g., "10 Mar – 16 Mar 2026") updates dynamically
- **Today highlight** — the current day column receives a ring highlight for quick orientation
- **"Month" button** — returns to the monthly grid from any week view

**Dynamic teaser count in month view**
- **ResizeObserver-based calculation** — a `ResizeObserver` on the month grid measures actual cell height and calculates how many teasers fit per day: `floor((cellHeight - 56px overhead) / 18px per item)`
- **Responsive to screen size** — on 4K monitors with tall cells, 5+ teasers are shown per day; on smaller screens, 1–2; the "+N more" count adjusts automatically
- **Viewport-filling grid** — both month and week grids use `flex-1` with `min-height: calc(100vh - 280px)` to fill the available viewport, eliminating wasted whitespace below the calendar
</Update>

## Goal-driven search & bookmarking

**Goal-based content filtering**
- **All Goals active by default** — the Discover page now uses the user's entire goal portfolio to filter content on initial load, combining all content_queries into a single batched Elasticsearch `bool.should` query for sub-100ms performance
- **Six search modes** — All Goals (default), Goal Group (category tab), Single Goal, Browse All (unfiltered), Random/Surprise, and Followed Publishers Only (toggle filter)
- **Batched ES query architecture** — instead of one HTTP request per content_query (which would mean 150+ requests for 50 goals), all queries are sent in a single request to the Discovery Node's new `POST /search/by-goals` endpoint
- **Random/Surprise mode** — "✨ Random" button uses Elasticsearch `function_score` with `random_score` for serendipitous content discovery, with optional light bias from user's goal categories
- **"All" button** — browse all content without goal filtering, using manual facet filters and full-text search
- **Content_queries resolved at portfolio load** — goal definitions are fetched from the Context Directory once (cached in Redis for 30 minutes), `{{param}}` placeholders substituted with user values, and attached to each goal for instant use in searches

**Bookmark popover**
- **Collection selector** — clicking the bookmark icon on any card opens a centered modal dialog showing all bookmark collections with checkboxes
- **Last-used collection pre-selected** — the most recently used collection is automatically checked for new bookmarks (persisted in localStorage)
- **Inline collection creation** — "New collection" button reveals an inline text input; created collections immediately appear in the list and get selected
- **Update & remove** — already-bookmarked teasers show "Update" and "Remove bookmark" options
- **Teaser preview** — the dialog shows the content title, type, and publisher for confirmation

**Discover UX improvements**
- **Scrollable facet lists** — filter groups with more than 8 options are now height-limited (240px) with overflow scrolling and a search-as-you-type input that filters by label or key
- **Publication date filter** — new dropdown with preset timespans (Today, Last 7 days, Last 30 days, This month, This quarter, This year) and custom from/to date pickers with locale-aware formatting
- **DN request debug panel** — collapsible `<details>` block below search results showing the exact JSON payloads sent to each Discovery Node, for Elasticsearch query debugging during development
- **Goal tab filter fix** — resolved "property filters should not exist" validation error when clicking goal tabs by adding `filters` and `contentQueries` to `GoalSearchDto`
- **Filter path mapping** — goal catalog paths (`category.tags`, `category.categories`) correctly mapped to DN classification paths (`classification.tags`, `classification.categories`); content-type-specific fields like `event_type` are ignored since the teaser generator doesn't populate them
</Update>
