## Overview

roadbeat Studio uses two authentication mechanisms:

- **JWT tokens** — for the Management API (admin UI, internal tools)
- **API keys** — for the Delivery API (websites, mobile apps, third-party consumers)

## JWT Authentication (Management API)

### Login

Obtain an access token and refresh token by sending your credentials:

```bash
POST /api/v1/auth/login
Content-Type: application/json

{
  "email": "admin@example.com",
  "password": "yourPassword"
}
```

**Response:**

```json
{
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIs...",
    "refreshToken": "eyJhbGciOiJIUzI1NiIs...",
    "expiresIn": 86400,
    "user": {
      "id": "user-uuid",
      "email": "admin@example.com",
      "name": "Admin User",
      "role": "admin",
      "organizationId": "org-uuid"
    }
  }
}
```

<Callout kind="info">
  The login endpoint is rate-limited to **5 requests per minute** per IP to prevent brute-force attacks.
</Callout>

### Account Lockout

Studio automatically locks a user account after **10 consecutive failed login attempts** within a 15-minute window. When an account is locked:

- Login returns `401 Unauthorized` with the message "Account is locked. Contact an administrator."
- The user's status is set to `locked` in the database
- An administrator must manually unlock the account by changing the status back to `active`

<Callout kind="alert">
  Account lockout is per-user, not per-IP. Even if the attacker switches IPs, the account remains locked after 10 failures. This prevents credential stuffing attacks.
</Callout>

### Using the Access Token

Include the JWT in the `Authorization` header for all Management API requests:

```bash
curl -X GET http://localhost:3000/api/v1/manage/content \
  -H "Authorization: Bearer eyJhbGciOiJIUzI1NiIs..."
```

### Token Expiration

| Token | Default Lifetime | Configurable |
|-------|-----------------|--------------|
| **Access token** | 24 hours | `JWT_EXPIRATION` env var |
| **Refresh token** | 7 days | `JWT_REFRESH_EXPIRATION` env var |

### Refreshing Tokens

When the access token expires, use the refresh token to obtain new tokens:

```bash
POST /api/v1/auth/refresh
Content-Type: application/json

{
  "refreshToken": "eyJhbGciOiJIUzI1NiIs..."
}
```

**Response:** Same format as the login response with fresh tokens.

<Callout kind="alert">
  Refresh tokens are single-use. Each refresh request returns a new refresh token, and the previous one is invalidated.
</Callout>

### JWT Payload

The access token contains:

```json
{
  "sub": "user-uuid",
  "email": "admin@example.com",
  "name": "Admin User",
  "role": "admin",
  "organizationId": "org-uuid",
  "iat": 1706100000,
  "exp": 1706186400
}
```

### Initial Setup

On a fresh installation, the first request should check if setup is needed:

```bash
GET /api/v1/auth/setup/status
```

If `needsSetup` is `true`, create the first admin account:

```bash
POST /api/v1/auth/setup
{
  "email": "admin@example.com",
  "password": "securePassword123!",
  "name": "Admin User",
  "organizationName": "My Organization"
}
```

<Callout kind="alert">
  The setup endpoint only works when no organizations exist in the database. After the first admin is created, it returns `409 Conflict`.
</Callout>

### Password Policy

All password entry points enforce the same policy. Passwords must meet **all** of the following requirements:

| Requirement | Description |
|-------------|-------------|
| **Minimum length** | At least 8 characters |
| **Uppercase letter** | At least one uppercase letter (A–Z) |
| **Lowercase letter** | At least one lowercase letter (a–z) |
| **Digit** | At least one number (0–9) |

If the password does not meet the policy, the API returns `400 Bad Request` with error code `WEAK_PASSWORD` and a list of the specific violations.

This policy is enforced on:
- Initial setup (`POST /auth/setup`)
- Password change (`POST /auth/change-password`)
- Password reset (`POST /auth/password/reset`)
- User creation by admin (`POST /manage/users`)

### Password Management

```bash
# Change password (requires current password)
POST /api/v1/auth/change-password
Authorization: Bearer YOUR_TOKEN
{
  "currentPassword": "oldPassword",
  "newPassword": "newSecurePassword123!"
}

# Request password reset (sends email with reset link)
POST /api/v1/auth/password/forgot
{ "email": "user@example.com" }

# Reset password with token (from email link)
POST /api/v1/auth/password/reset
{
  "token": "reset-token-from-email",
  "newPassword": "newSecurePassword123!"
}
```

<Callout kind="info">
  Password reset emails are sent via the configured mail transport. In development (`MAIL_TRANSPORT=log`), the reset link is printed to the server console. In production, configure SMTP — see [Configuration](/deployment/configuration).
</Callout>

### Logout

Revoke the current refresh token:

```bash
POST /api/v1/auth/logout
Authorization: Bearer YOUR_TOKEN
Content-Type: application/json

{
  "refreshToken": "eyJhbGciOiJIUzI1NiIs..."
}
```

### Session Management

View and manage active sessions (refresh tokens) for the current user:

```bash
# List active sessions
GET /api/v1/auth/sessions
Authorization: Bearer YOUR_TOKEN

# Revoke a specific session
DELETE /api/v1/auth/sessions/:id
Authorization: Bearer YOUR_TOKEN

# Revoke all sessions (logout everywhere)
DELETE /api/v1/auth/sessions
Authorization: Bearer YOUR_TOKEN
```

Each session includes `ip`, `userAgent`, `createdAt`, and `lastUsedAt` for transparency.

<Callout kind="alert">
  Refresh tokens are now persisted server-side and are single-use. If a previously consumed refresh token is replayed (token reuse detection), all sessions for the user are immediately revoked as a security precaution.
</Callout>

## Two-Factor Authentication (MFA)

Studio supports TOTP-based two-factor authentication, compatible with Google Authenticator, Authy, 1Password, and other TOTP apps.

### Check MFA Status

```bash
GET /api/v1/auth/mfa/status
Authorization: Bearer YOUR_TOKEN
```

**Response:**

```json
{ "data": { "enabled": false } }
```

### Enroll in MFA

<Steps>
  <Step title="Start enrollment">
    ```bash
    POST /api/v1/auth/mfa/enroll
    Authorization: Bearer YOUR_TOKEN
    ```

    **Response:**

    ```json
    {
      "data": {
        "secret": "JBSWY3DPEHPK3PXP",
        "otpauthUrl": "otpauth://totp/roadbeat:user@example.com?secret=...",
        "qrCodeDataUrl": "data:image/png;base64,iVBOR..."
      }
    }
    ```
  </Step>
  <Step title="Scan QR code">
    Scan the QR code (or manually enter the secret) in your authenticator app.
  </Step>
  <Step title="Verify enrollment">
    Submit a code from your authenticator to complete enrollment:

    ```bash
    POST /api/v1/auth/mfa/verify
    Authorization: Bearer YOUR_TOKEN
    Content-Type: application/json

    { "code": "123456" }
    ```

    **Response:**

    ```json
    {
      "data": {
        "enabled": true,
        "recoveryCodes": [
          "A1B2-C3D4-E5F6",
          "G7H8-I9J0-K1L2",
          "..."
        ]
      }
    }
    ```
  </Step>
</Steps>

<Callout kind="danger">
  Save your recovery codes in a secure location. Each code can only be used **once**, and they are the only way to regain access if you lose your authenticator device.
</Callout>

### Login with MFA

When MFA is enabled, login requires a two-step flow:

```bash
# Step 1: Login with credentials (no mfaCode)
POST /api/v1/auth/login
{ "email": "user@example.com", "password": "yourPassword" }

# Response: MFA challenge
{
  "data": {
    "mfaRequired": true,
    "mfaToken": "eyJhbG...",
    "accessToken": null,
    "refreshToken": null
  }
}
```

```bash
# Step 2: Login again with credentials + mfaCode
POST /api/v1/auth/login
{
  "email": "user@example.com",
  "password": "yourPassword",
  "mfaCode": "123456"
}
```

The `mfaCode` field accepts either a 6-digit TOTP code or a recovery code in `XXXX-XXXX-XXXX` format.

### Disable MFA

```bash
POST /api/v1/auth/mfa/disable
Authorization: Bearer YOUR_TOKEN
Content-Type: application/json

{
  "code": "123456",
  "password": "yourPassword"
}
```

### Regenerate Recovery Codes

```bash
POST /api/v1/auth/mfa/recovery-codes
Authorization: Bearer YOUR_TOKEN
Content-Type: application/json

{ "code": "123456" }
```

<Callout kind="info">
  Regenerating recovery codes invalidates all previous codes. A valid TOTP code (not a recovery code) is required.
</Callout>

## API Key Authentication (Delivery API)

### Creating API Keys

API keys are created by admins via the Management API or the Settings UI:

```bash
POST /api/v1/manage/api-keys
Authorization: Bearer YOUR_JWT_TOKEN
{
  "name": "Production Website",
  "permissions": ["content.read", "content-types.read", "assets.read"]
}
```

**Response:**

```json
{
  "data": {
    "id": "key-uuid",
    "name": "Production Website",
    "key": "rbk_a1b2c3d4e5f6...",
    "permissions": ["content.read", "content-types.read", "assets.read"],
    "createdAt": "2026-01-24T10:00:00Z"
  }
}
```

<Callout kind="danger">
  The full API key (`rbk_...`) is only shown once when created. Store it securely — it cannot be retrieved later.
</Callout>

<Callout kind="info">
  CE supports up to **2 API keys** per organization. Pro has unlimited API keys.
</Callout>

### Using API Keys

Pass the API key in the `Authorization` header or `X-Api-Key` header:

<Tabs>
  <Tab title="Authorization Header" icon="lock">
    ```bash
    curl -X GET http://localhost:3000/api/v1/delivery/content \
      -H "Authorization: Bearer rbk_a1b2c3d4e5f6..."
    ```
  </Tab>
  <Tab title="X-Api-Key Header" icon="key">
    ```bash
    curl -X GET http://localhost:3000/api/v1/delivery/content \
      -H "X-Api-Key: rbk_a1b2c3d4e5f6..."
    ```
  </Tab>
</Tabs>

### API Key Permissions

Each API key has scoped permissions:

| Permission | Grants Access To |
|------------|-----------------|
| `content.read` | List and get published content |
| `content-types.read` | List and get content type schemas |
| `assets.read` | List and get asset metadata |

### Managing API Keys

```bash
# List API keys
GET /api/v1/manage/api-keys

# Delete an API key
DELETE /api/v1/manage/api-keys/:id
```

## Security Best Practices

- **Never expose JWT tokens** in client-side code or URLs
- **Store API keys** in environment variables, not in source code
- **Use HTTPS** in production to encrypt tokens in transit
- **Rotate API keys** periodically and when team members leave
- **Use scoped permissions** — only grant the minimum permissions needed
- **Enable MFA** for all admin accounts — TOTP adds a strong second factor against credential theft
- **Store recovery codes securely** — treat them like passwords; they bypass TOTP
- **Monitor for 401 errors** — may indicate compromised credentials
- **Configure SMTP** for production so password reset emails are delivered (not just logged)
- **Review login attempts** — failed login attempts are recorded in the `login_attempts` table for auditing
- **Use session management** — periodically review active sessions via `GET /auth/sessions` and revoke unfamiliar ones

## Common Authentication Errors

<Expandable title="401 Unauthorized — Missing or invalid token" default-open="false">
  - Verify the `Authorization` header is present
  - Check that the token hasn't expired
  - Ensure the token is prefixed with `Bearer ` (with a space)
</Expandable>

<Expandable title="403 Forbidden — Insufficient permissions" default-open="false">
  - Check the user's role (admin, editor, viewer)
  - For API keys, verify the key has the required permission scope
  - Admin-only endpoints return 403 for editor/viewer roles
</Expandable>

<Expandable title="401 Unauthorized — Account is locked" default-open="false">
  - The account was locked after 10 consecutive failed login attempts within 15 minutes
  - Contact an administrator to unlock the account (set status back to `active`)
  - The lockout protects against brute-force and credential stuffing attacks
</Expandable>

<Expandable title="400 Bad Request — Weak password (WEAK_PASSWORD)" default-open="false">
  - The password does not meet the password policy requirements
  - Check the `message` field for a list of specific violations
  - Password must be at least 8 characters with uppercase, lowercase, and a digit
</Expandable>

<Expandable title="401 Unauthorized — Invalid MFA code" default-open="false">
  - The TOTP code is incorrect or has expired (codes rotate every 30 seconds)
  - If using a recovery code, check that it hasn't already been used — each recovery code is single-use
  - Ensure your authenticator app's clock is synced (TOTP depends on accurate time)
</Expandable>

<Expandable title="429 Too Many Requests — Rate limit exceeded" default-open="false">
  - Wait for the rate limit window to reset (check `X-RateLimit-Reset` header)
  - The login endpoint has a stricter limit of 5 requests/minute
  - The forgot-password endpoint is limited to 3 requests/minute
  - Consider implementing exponential backoff in your client
</Expandable>
