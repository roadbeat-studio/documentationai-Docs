## Overview

roadbeat Studio provides role-based access control (RBAC) to manage who can do what within your organization. CE includes three organization-level roles plus a cross-org **Super Admin** role for platform administration. Pro plugins extend this with granular per-content-type permissions and custom roles.

## User Management

### Creating Users

<Tabs>
  <Tab title="Admin UI" icon="monitor">
    1. Navigate to **Users** in the sidebar (admin only)
    2. Click **Add User**
    3. Enter email, name, and select a role
    4. The user receives an invitation and sets their password
  </Tab>
  <Tab title="API" icon="code">
    ```bash
    POST /api/v1/manage/users
    Content-Type: application/json
    Authorization: Bearer YOUR_JWT_TOKEN

    {
      "email": "editor@example.com",
      "name": "Jane Editor",
      "role": "editor",
      "password": "securePassword123!"
    }
    ```
  </Tab>
</Tabs>

<Callout kind="alert">
  CE defaults to **3 users** per organization. This limit can be configured per-org by a super admin via the Platform Admin dashboard. Pro has unlimited users.
</Callout>

### User Properties

| Field | Description |
|-------|-------------|
| **Email** | Unique login identifier |
| **Name** | Display name |
| **Role** | One of: `admin`, `editor`, `viewer` |
| **Status** | `active`, `inactive`, `suspended`, or `locked` |
| **Avatar URL** | Optional profile image |
| **Last Login** | Timestamp of most recent login |

### Updating Users

Admins can update any user's profile, role, or status:

```bash
PUT /api/v1/manage/users/:id
{
  "name": "Jane Senior Editor",
  "role": "admin"
}
```

### Deleting Users

```bash
DELETE /api/v1/manage/users/:id
```

<Callout kind="alert">
  You cannot delete the last admin user in an organization.
</Callout>

## Roles

### CE Roles

| Role | Content | Assets | Users | Settings | Publishing |
|------|---------|--------|-------|----------|------------|
| **Admin** | Full CRUD | Full CRUD | Manage | Full access | Publish/Unpublish |
| **Editor** | Create, Edit, Publish | Upload, Edit | View own | Read only | Publish/Unpublish |
| **Viewer** | Read only | Read only | — | — | — |

### Role Assignment

Each user is assigned exactly one role. The role determines which API endpoints and UI features are accessible.

Roles are enforced at two levels:

1. **API level** — a guard checks the required role/permission on each controller method
2. **UI level** — the Angular frontend conditionally shows/hides elements based on the user's role

#### Permission model

Under the hood, authorization is moving from coarse role checks to a
`resource:action` **permission registry** (e.g. `content:publish`, `asset:write`,
`user:unlock`, `audit:read`). The three CE roles map to sensible permission sets —
**admin** holds everything, **editor** can author and publish content and manage
assets (but not delete content or manage users), **viewer** is read-only. This is
the foundation the Pro `pro-advanced-roles` plugin builds custom roles on.

## Authentication

### Login Flow

<Steps>
  <Step title="POST /api/v1/auth/login">
    Send email and password. Receive an access token (JWT) and refresh token.

    ```bash
    POST /api/v1/auth/login
    { "email": "admin@example.com", "password": "yourPassword" }
    ```
  </Step>
  <Step title="Use access token">
    Include the JWT in the `Authorization` header for all subsequent requests.

    ```bash
    Authorization: Bearer eyJhbGciOiJIUzI1NiIs...
    ```
  </Step>
  <Step title="Refresh when expired">
    When the access token expires (default: 24h), use the refresh token to get new tokens.

    ```bash
    POST /api/v1/auth/refresh
    { "refreshToken": "your-refresh-token" }
    ```
  </Step>
</Steps>

### Initial Setup

On first run, Studio requires an initial admin setup:

```bash
# Check if setup is needed
GET /api/v1/auth/setup/status
# Returns: { "data": { "needsSetup": true } }

# Perform setup
POST /api/v1/auth/setup
{
  "email": "admin@example.com",
  "password": "securePassword123!",
  "name": "Admin User",
  "organizationName": "My Organization"
}
```

### Password Policy

All passwords must meet the following requirements:

- At least **8 characters** (configurable — `PASSWORD_MIN_LENGTH`)
- At least **one uppercase letter** (A–Z)
- At least **one lowercase letter** (a–z)
- At least **one digit** (0–9)
- Optionally **one symbol** — enable with `PASSWORD_REQUIRE_SYMBOL=true`

The policy is enforced declaratively at every entry point (initial setup, user
creation, password change, password reset). Raising the minimum length or
requiring a symbol takes effect without any code changes.

### Password Management

- **Change password** (requires current password): `POST /api/v1/auth/change-password`
- **Forgot password**: `POST /api/v1/auth/password/forgot` — sends a password reset email
- **Reset password**: `POST /api/v1/auth/password/reset` — validates token and sets new password

<Callout kind="info">
  Password reset tokens are single-use and expire after 1 hour. The forgot-password endpoint always returns success to prevent email enumeration attacks. In development mode (`MAIL_TRANSPORT=log`), the reset link is printed to the server console.
</Callout>

### Account Lockout, Unlock & Captcha

After **10 consecutive failed login attempts** within 15 minutes, the account is
automatically locked (`status = 'locked'`). Recover it without touching the
database:

```bash
# Admin unlocks a locked user (org-scoped) — resets status + clears failed attempts
POST /api/v1/manage/users/:id/unlock
```

- **Auto-unlock** — set `LOCKOUT_AUTO_UNLOCK_MS` so a lock older than the window
  clears itself on the next login attempt.
- **Captcha** — set `CAPTCHA_PROVIDER` (`turnstile` or `hcaptcha`) + `CAPTCHA_SECRET`
  to require a captcha on login and password-reset. No-op when unset; verification
  fails closed if the captcha service is unreachable.

<Callout kind="alert">
  Login attempts are tracked in the `login_attempts` table (email, IP,
  success/failure, timestamp). Unlock is strictly org-scoped — an admin of one
  organization cannot unlock a user in another.
</Callout>

## Profile Management

Users can manage their own profile:

```bash
# Get own profile
GET /api/v1/auth/me

# Update own profile
PATCH /api/v1/auth/me
{ "name": "New Name", "avatarUrl": "https://..." }
```

## GDPR — Export & Erase a User's Data

Org admins can honour data-subject requests in-product, strictly org-scoped:

```bash
# Art. 15 — export a portable JSON bundle of the user's personal data
GET /api/v1/manage/users/:id/export

# Art. 17 — erase (anonymize) the user; deletes auth artifacts + user-generated PII
POST /api/v1/manage/users/:id/erase
```

<Callout kind="info">
  **Retention policy — anonymize, not delete.** Erasure anonymizes the user row
  (strips email/name/password/avatar, `status = deleted`) so authored content
  keeps referential integrity, while auth artifacts and user-generated data
  (sessions, MFA secret, login attempts, password resets, SSO tokens,
  notifications, comments) are hard-deleted. The last active admin cannot be
  erased. Both operations emit audit events.
</Callout>

## Audit Log (Community Edition)

A self-hostable, **append-only** who-changed-what-when trail records
security-relevant events (user create/update/delete, login, unlock, erase, org
changes). Secrets are redacted before storage.

```bash
# Org-scoped, paginated, filterable by action / actor / resource / date
GET /api/v1/manage/audit?action=user.updated&limit=50
```

Retention is unlimited by default; set `CE_AUDIT_RETENTION_DAYS` to prune older
entries. Pro `pro-audit` remains the richer, immutable-storage superset.

## Per-Org Quotas & Usage

Every quota-bearing resource (users, webhooks, API keys, locales, discovery
nodes, storage) is enforced against the organization's **effective limit**
(org override → license → CE default). Inspect current usage:

```bash
GET /api/v1/manage/usage
# → per-resource { used, limit } (limit -1 = unlimited)
```

Super admins set per-org limits from the Platform Admin dashboard; `-1` means
unlimited (Pro tiers).

## Super Admin & Platform Administration

Super admins operate at the **platform level**, above individual organizations. They can manage all organizations, view cross-org statistics, and grant or revoke super admin privileges.

### Becoming a Super Admin

The first super admin is created automatically during the initial setup wizard. Additional super admins can be granted by existing super admins via:

- **Platform Admin UI** — navigate to **Platform** → **Admins** → search for a user → click **Grant Super Admin**
- **API** — `POST /api/v1/platform/admins/:userId/grant`

### Platform Admin Dashboard

Super admins see a **Platform Admin** link in the sidebar (shield icon). The dashboard provides:

- **Organization list** — search, filter by plan/status, paginate, create new orgs
- **Organization detail** — view users, content stats, settings; update plan and limits
- **Platform statistics** — total orgs, users, content items, storage usage; orgs-by-plan chart
- **Admin management** — list current super admins, grant/revoke privileges

<Callout kind="alert">
  Super admin access is tracked in the `platform_admins` table. Revoking super admin does not affect the user's organization-level role.
</Callout>

### Organization Switching

Super admins can switch their active organization context without logging out. This re-issues a new JWT scoped to the target organization, allowing the super admin to manage content and users within that org.

- **Platform Admin UI** — use the organization switcher in the sidebar to select a different org
- **API** — `POST /api/v1/platform/organizations/:id/switch` — returns new access and refresh tokens scoped to the target organization

<Callout kind="info">
  Switching org context does not change the super admin's home organization or their org-level role. The new JWT simply carries a different `organizationId` claim.
</Callout>

### Multi-Org Login

If the same email address exists in multiple organizations, the login response returns an **org picker** instead of a JWT. The user selects which organization to log into, then receives tokens scoped to that organization.

## Pro: Advanced Roles

With the `pro-advanced-roles` plugin, you get:

- **Per-content-type permissions** — different access levels for different content types
- **Priority-based resolution** — user-specific rules override role-based rules
- **Allow/Deny rules** — explicitly grant or deny specific actions
- **Permission matrix** — visual overview of who can do what

## Pro: SSO Integration

With the `pro-sso` plugin, you get:

- **SAML 2.0** — integrate with enterprise identity providers
- **OIDC** — OpenID Connect for modern SSO
- **SCIM** — automatic user provisioning and deprovisioning
