## Overview

Publishing in roadbeat Studio is a three-step process:

1. **Publishing locally** — setting content status to `published`, making it available via the Delivery API
2. **Distributing to Discovery Nodes** — pushing a cryptographically signed teaser to the search network
3. **Delivering to Content Pod** — bundling the full content item for self-hosted pod delivery (if configured)

Steps 2 and 3 are triggered automatically when you publish from the Content Editor.

## Publishing Pipeline

When you publish content, Studio executes this pipeline:

<Steps>
  <Step title="Validate" icon="check-circle">
    The content is validated against its JSON Schema. Required fields are checked, and reference integrity is verified. If validation fails, the publish is blocked with a detailed error.
  </Step>
  <Step title="Generate Teaser" icon="file-text">
    A teaser is generated from the full content using the content type's **teaser schema**. The teaser contains only the fields needed for search and preview — title, description, classification, temporal and geographic data. Additionally, **content-type-specific filter values** are extracted from `x-filter` schema annotations into `teaser.type_filters` (see [Type Filters](#type-filters) below).
  </Step>
  <Step title="Sign with Ed25519" icon="key">
    The teaser is signed with the publisher's **Ed25519 private key**. Discovery Nodes verify this signature to ensure content authenticity and prevent tampering.
  </Step>
  <Step title="Distribute" icon="send">
    The signed teaser is pushed to all configured **Discovery Nodes**. Each node indexes the teaser for search and serves it to consumers.
  </Step>
  <Step title="Verify" icon="shield">
    Studio confirms receipt from each Discovery Node and records the publish status. Failed deliveries are retried automatically.
  </Step>
</Steps>

## Discovery Nodes

Discovery Nodes are the search infrastructure of the roadbeat network. You configure which nodes to publish to in **Settings → Discovery Nodes**.

### Adding a Discovery Node

```bash
POST /api/v1/manage/discovery-nodes
{
  "name": "EU News Node",
  "endpoint": "https://news-eu-1.roadbeat.net",
  "contentTypes": ["news", "press-releases"],
  "enabled": true
}
```

### Publish Status

Check the publish status of any content item:

```bash
GET /api/v1/manage/publishing/content/:id/status
```

Returns the status per Discovery Node:

```json
{
  "data": {
    "contentId": "uuid",
    "nodes": [
      {
        "nodeId": "news-eu-1",
        "status": "published",
        "publishedAt": "2026-01-24T14:30:00Z",
        "teaserId": "teaser-uuid"
      },
      {
        "nodeId": "news-eu-2",
        "status": "failed",
        "error": "Connection timeout",
        "lastAttempt": "2026-01-24T14:30:05Z"
      }
    ]
  }
}
```

## Publisher Keys

Studio uses **Ed25519 key pairs** for teaser signing. Discovery Nodes use the public key to verify that teasers genuinely came from your Studio instance.

### Key Management

```bash
# List publisher keys (public keys only — private keys are never exposed)
GET /api/v1/manage/publishing/keys

# Get a specific public key
GET /api/v1/manage/publishing/keys/:id/public

# Rotate keys (generates new key, revokes previous)
POST /api/v1/manage/publishing/keys/rotate
```

<Callout kind="alert">
  Rotating keys invalidates all previously signed teasers on Discovery Nodes. Only rotate if your private key has been compromised.
</Callout>

## Teaser Preview

Preview the teaser that would be generated for a content item without actually publishing:

```bash
GET /api/v1/manage/publishing/content/:id/teaser-preview
```

This is useful for verifying that the teaser schema extracts the right fields before distributing.

## Retry Failed Publishes

If distribution to a Discovery Node fails, you can retry manually:

```bash
POST /api/v1/manage/publishing/content/:id/retry/:nodeId
```

Studio also supports automatic retries with configurable backoff:

```bash
# Default retry config
PUBLISHING_MAX_RETRIES=3
PUBLISHING_BACKOFF_MS=1000,5000,30000
```

## Withdrawing Content

To remove content from Discovery Nodes (unpublish from the network):

```bash
POST /api/v1/manage/publishing/content/:id/withdraw
```

This sends an unpublish request to all nodes that have the teaser.

## Automatic Publishing

The publishing pipeline is triggered automatically when:

- Content status changes to `published` via the UI or API
- Content is updated while in `published` status (re-distributes the updated teaser)
- Content is unpublished or deleted (sends withdrawal to Discovery Nodes)

This is handled by the `PublishingEventListener` which listens to `content.afterPublish`, `content.afterUnpublish`, and `content.afterDelete` events.

## Type Filters

When generating a teaser, Studio automatically extracts **content-type-specific filter values** from the content data using `x-filter` annotations in the metadata schema. These values are stored in `teaser.type_filters` and indexed by Discovery Nodes for faceted filtering.

### How it works

The `TeaserGeneratorService` fetches the content type's metadata schema from the Schema Registry and walks its properties recursively:

<Steps>
  <Step title="Fetch schema" icon="download">
    The metadata schema is fetched from the Schema Registry (e.g., `news-metadata.schema.json`) which contains `x-filter` annotations on filterable properties.
  </Step>
  <Step title="Walk properties" icon="search">
    The `walkSchemaForFilters()` method recursively traverses schema properties, looking for `x-filter.enabled: true` annotations. It resolves values from both **nested object paths** (e.g., `data.editorial.is_breaking`) and **flat dot-notation keys** (e.g., `data["editorial.is_breaking"]`) for backwards compatibility.
  </Step>
  <Step title="Extract values" icon="filter">
    Values are extracted based on their type: simple strings/numbers/booleans are passed through directly, objects use `value_path` for nested extraction, and arrays handle mixed content (plain strings + objects with value_path).
  </Step>
  <Step title="Attach to teaser" icon="package">
    Extracted values are stored in `teaser.type_filters` as a flat key-value map (e.g., `{ is_breaking: true, article_type: "analysis", editorial_priority: "normal" }`).
  </Step>
</Steps>

### Boolean values

Boolean fields (e.g., `is_breaking`, `is_exclusive`, `is_free`) are stored as native ES booleans (`true`/`false`). The Discovery Node's query builder automatically coerces frontend string values (`"1"`, `"0"`) to proper booleans when building Elasticsearch queries.

<Callout kind="tip">
  To add new filters to any content type, add an `x-filter` annotation to the property in its metadata schema. No code changes are needed — the enrichment pipeline picks up new annotations automatically.
</Callout>

### Republishing

After schema changes or code updates affecting filter extraction, use the bulk republish endpoint to re-index all content:

```bash
POST /api/v1/manage/publishing/republish-all
```

This regenerates teasers for all published content and redistributes them to Discovery Nodes.

## Content Pod Delivery

In addition to Discovery Node distribution, published content can be delivered to a **Content Pod** — a file-based bundle written to disk and served from a static or dynamic server.

### Automatic Pod Publish

If your organization has an active Content Pod configured, the pod publish runs automatically every time you publish from the Content Editor:

1. Content status is set to `published` in Studio
2. Studio checks for an active pod configuration
3. `POST /api/v1/manage/content-pod/publish/:contentId` is called automatically
4. The bundle (JSON, HTML, embed fragment) is written to the pod filesystem
5. The pod `manifest.json` is updated atomically

You will see two toast notifications: **"Content published"** and **"Published to Content Pod"**.

<Callout kind="tip">
  A Content Pod publish failure does not block the Studio publish. The content remains available via the Delivery API. Retry the pod publish manually from **Settings → Content Pod**.
</Callout>

### Viewing Published Pod Content

Navigate to **Settings → Content Pod → Dashboard → Content tab** to see all items currently in the pod manifest — title, locale, content type, size, and a direct link to each item's pod URL.

For full documentation on pod setup, modes, quotas, and the pod manifest API, see the [Content Pod guide](/guides/content-pod).
