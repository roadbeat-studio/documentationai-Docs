## Overview

roadbeat Studio provides built-in backup, export, and import capabilities to ensure data portability, disaster recovery, and smooth migration between instances. You can export your entire database, generate Content Pod bundles, and import data from other Studio instances with full conflict resolution.

<Callout kind="tip">
  For a guided experience, use the **Migration Wizard** at **Settings → Organization → Open Migration Wizard** or navigate directly to `/settings/migration`. See the [Migration guide](/guides/migration) for details.
</Callout>

## Database Backup

### Full Backup

Create a full backup archive (ZIP) of content, content types, assets metadata, and settings:

```bash
POST /api/v1/export/backups
{
  "label": "Pre-migration backup",
  "includeContent": true,
  "includeAssets": true,
  "includeConfig": true
}
```

This creates a backup containing:

- All content items (with version history)
- All content types and schemas
- All assets (metadata only — files are in S3)
- All users and organization settings
- Webhook configurations
- API keys (hashed)

<Callout kind="alert">
  Backups contain sensitive data including hashed passwords and API key hashes. Store them securely and restrict access.
</Callout>

### Export Progress

During export, you can poll the progress of an in-progress backup:

```bash
GET /api/v1/export/backups/:id/progress
```

The response includes the current phase, percentage, and a human-readable message:

```json
{
  "data": {
    "backupId": "abc-123",
    "phase": "content",
    "percentage": 50,
    "message": "Exported 142 content items",
    "itemsProcessed": 142,
    "itemsTotal": 142
  }
}
```

Export phases progress through: `preparing` → `content_types` → `content` → `assets` → `config` → `archiving` → `storing` → `completed`.

### Listing and Downloading Backups

```bash
# List all backups
GET /api/v1/export/backups

# Get backup status
GET /api/v1/export/backups/:id

# Download backup archive (ZIP)
GET /api/v1/export/backups/:id/download

# Delete a backup
DELETE /api/v1/export/backups/:id
```

### Restoring from Backup

```bash
POST /api/v1/export/backups/:id/restore
{
  "restoreContent": true,
  "restoreAssets": true,
  "restoreConfig": true
}
```

<Callout kind="danger">
  Restoring a backup **replaces existing data** for the selected categories. This action cannot be undone. Always create a fresh backup before restoring.
</Callout>

## Import with Validation & Conflict Resolution

When importing data from another Studio instance, you can validate the archive first and choose how conflicts are handled.

### Validating an Archive

Upload a ZIP file for pre-import validation. The endpoint checks the manifest, counts items, and detects conflicts with existing data:

```bash
POST /api/v1/export/validate-import
Content-Type: multipart/form-data

file: <backup.zip>
```

The response includes:

```json
{
  "data": {
    "valid": true,
    "manifestVersion": "1.0",
    "generator": "roadbeat-studio",
    "sourceOrgId": "org-abc-123",
    "counts": {
      "contentTypes": 5,
      "content": 142,
      "assets": 38
    },
    "conflicts": [
      {
        "type": "content",
        "id": "item-456",
        "name": "welcome-post",
        "existingUpdatedAt": "2026-04-10T12:00:00Z",
        "incomingUpdatedAt": "2026-04-15T08:30:00Z"
      }
    ],
    "errors": []
  }
}
```

### Importing with Conflict Resolution

Import the archive with a chosen conflict resolution strategy:

```bash
POST /api/v1/export/import
Content-Type: multipart/form-data

file: <backup.zip>
strategy: "merge"
restoreContent: true
restoreAssets: true
restoreConfig: true
```

Three conflict resolution strategies are available:

| Strategy | Behavior |
|---|---|
| `skip` | Keep existing items unchanged. Only new items are imported. |
| `overwrite` | Replace existing items with imported data. |
| `merge` | Compare timestamps — the more recently updated version wins. |

<Callout kind="info">
  The `merge` strategy is recommended for most migration scenarios. It preserves recent edits on the target instance while still importing newer changes from the source.
</Callout>

## Content Pod Export

Generate a **Content Pod** — a portable archive of published content with JSON files, manifest, and HTML index:

```bash
POST /api/v1/export/pods
{
  "contentTypeId": "news",
  "status": "published",
  "label": "News Export Q1 2026"
}
```

The Content Pod bundle includes:

- Content data as JSON files
- HTML rendered pages (via templates)
- Referenced asset files
- A `manifest.json` describing the bundle
- Content integrity signatures

<Callout kind="info">
  Content Pods are a key part of the roadbeat ecosystem, enabling individuals and small publishers to host content without running a full Studio instance. See the Content Pod documentation for details.
</Callout>

## Pro: Advanced Export

With the `pro-export` plugin, you get additional capabilities:

- **CSV export** with configurable field mappings and transforms
- **Export templates** — reusable export configurations
- **Field transforms** — lowercase, uppercase, trim, date formatting, truncation
- **Scheduled exports** — automatic recurring exports
- **Export job management** — track export progress and download results

## Automated Backups

For production deployments, set up automated backups:

<Tabs>
  <Tab title="PostgreSQL" icon="database">
    Use `pg_dump` for native PostgreSQL backups:

    ```bash
    # Daily backup via cron
    pg_dump -U postgres roadbeat_studio > backup_$(date +%Y%m%d).sql
    ```
  </Tab>
  <Tab title="Docker" icon="server">
    If running via Docker Compose, backup the database volume:

    ```bash
    docker compose exec postgres pg_dump -U studio studio > backup.sql
    ```
  </Tab>
</Tabs>

<Callout kind="tip">
  For asset files stored in S3-compatible storage, use the provider's native backup tools (e.g., `aws s3 sync` or Hetzner Object Storage versioning).
</Callout>
