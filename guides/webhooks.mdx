## Overview

Webhooks allow Studio to send real-time HTTP notifications to external systems when events occur — such as content being published, updated, or deleted. Each webhook delivery is signed with HMAC-SHA256 so the receiver can verify authenticity.

<Callout kind="info">
  CE supports up to **3 webhooks** per organization. Pro has unlimited webhooks.
</Callout>

## Creating a Webhook

<Tabs>
  <Tab title="Admin UI" icon="monitor">
    1. Navigate to **Settings → Webhooks**
    2. Click **Add Webhook**
    3. Enter the target URL and select which events to listen for
    4. Save — Studio generates a **signing secret** for HMAC verification
  </Tab>
  <Tab title="API" icon="code">
    ```bash
    POST /api/v1/manage/webhooks
    {
      "url": "https://example.com/webhook",
      "events": ["content.afterPublish", "content.afterUpdate"],
      "enabled": true
    }
    ```
  </Tab>
</Tabs>

## Supported Events

| Event | Trigger |
|-------|---------|
| `content.afterCreate` | New content item created |
| `content.afterUpdate` | Content item updated |
| `content.afterDelete` | Content item deleted |
| `content.afterPublish` | Content published |
| `content.afterUnpublish` | Content unpublished |
| `asset.afterUpload` | New asset uploaded |
| `asset.afterDelete` | Asset deleted |
| `user.afterCreate` | New user created |
| `user.afterLogin` | User logged in |

## Payload Format

Every webhook delivery includes a JSON payload:

```json
{
  "event": "content.afterPublish",
  "timestamp": "2026-01-24T14:30:00Z",
  "data": {
    "contentId": "uuid",
    "contentTypeId": "news",
    "organizationId": "org-uuid",
    "userId": "user-uuid",
    "status": "published"
  }
}
```

## Signature Verification

Every delivery is signed with HMAC-SHA256 and carries these headers:

| Header | Description |
|--------|-------------|
| `X-Webhook-Signature` | `sha256=<hex>` — the HMAC over `<timestamp>.<rawBody>`. During a secret rotation this is a comma-separated list of signatures (verify against any). |
| `X-Webhook-Timestamp` | Unix seconds; included in the signed string so you can reject stale deliveries. |
| `X-Idempotency-Key` | Stable per-event id, constant across retries and replays — dedupe on it. |
| `X-Webhook-Event` | The event name. |

<Callout kind="alert">
  The signing secret is **never** sent in a header. Compute the HMAC over
  `<timestamp>.<rawBody>` with your stored secret and compare in constant time.
</Callout>

Verify it in your receiver (recommended timestamp tolerance: ±5 minutes):

<CodeGroup tabs="Node.js,Python">
```javascript
const crypto = require('crypto');

function verifyWebhook(rawBody, headers, secret) {
  const timestamp = headers['x-webhook-timestamp'];
  // Reject stale deliveries (replay protection).
  if (Math.abs(Date.now() / 1000 - Number(timestamp)) > 300) return false;

  const expected = 'sha256=' + crypto
    .createHmac('sha256', secret)
    .update(`${timestamp}.${rawBody}`)
    .digest('hex');

  // The header may hold several comma-separated signatures during rotation.
  return (headers['x-webhook-signature'] || '')
    .split(',')
    .some((sig) => {
      const a = Buffer.from(sig.trim());
      const b = Buffer.from(expected);
      return a.length === b.length && crypto.timingSafeEqual(a, b);
    });
}
```
```python
import hmac, hashlib, time

def verify_webhook(raw_body: bytes, headers: dict, secret: str) -> bool:
    timestamp = headers["x-webhook-timestamp"]
    if abs(time.time() - int(timestamp)) > 300:  # replay window
        return False

    signed = f"{timestamp}.".encode() + raw_body
    expected = "sha256=" + hmac.new(secret.encode(), signed, hashlib.sha256).hexdigest()

    # Multiple comma-separated signatures may be present during rotation.
    return any(
        hmac.compare_digest(sig.strip(), expected)
        for sig in headers.get("x-webhook-signature", "").split(",")
    )
```
</CodeGroup>

<Callout kind="tip">
  Always verify against the **raw request body bytes**, not a re-serialized JSON
  object — re-serialization can reorder keys and change the signature.
</Callout>

## Idempotency

Retries and manual replays reuse the same `X-Idempotency-Key`, so the same
logical event may arrive more than once. Record processed keys and skip
duplicates to make your handler safe to re-run.

## Rotating the Signing Secret

Rotate a webhook's secret without downtime — during the rotation window Studio
**dual-signs** every delivery with both the new and previous secret, so a
receiver holding either one still verifies:

```bash
POST /api/v1/manage/webhooks/:id/rotate-secret
# → { "secret": "<new>", "secretPrevious": "<old>" }
```

Update your receiver to the new secret; the previous secret keeps verifying
until the next rotation.

## Allowed Targets (SSRF protection)

Webhook URLs that resolve to private, loopback, link-local, or cloud-metadata
addresses (e.g. `169.254.169.254`, `localhost`, `10.0.0.0/8`) are **rejected** —
at registration **and** re-checked at delivery time. Self-hosted deployments
that genuinely need internal targets can opt in with
`WEBHOOK_ALLOW_PRIVATE_TARGETS=true`.

## Testing

Send a test payload to verify your webhook endpoint is reachable:

```bash
POST /api/v1/manage/webhooks/:id/test
```

This sends a test event with sample data to your configured URL.

## Delivery & Retries

Webhook deliveries are processed asynchronously via **BullMQ**. If a delivery fails (non-2xx response or timeout), Studio retries automatically:

- **Retry attempts**: 3 (configurable in Pro)
- **Backoff**: Exponential — 1s, 5s, 30s
- **Timeout**: 10 seconds per attempt

### Delivery Log, Failed Deliveries & Replay

Each webhook records every delivery (timestamp, HTTP status, response time,
success/failure). List deliveries — optionally just the failures:

```bash
# Recent deliveries for a webhook (add ?status=failed to see only failures)
GET /api/v1/manage/webhooks/:id/deliveries?status=failed

# Re-send a specific delivery — same X-Idempotency-Key, so the receiver
# still dedupes; a fresh attempt actually re-sends
POST /api/v1/manage/webhooks/deliveries/:deliveryId/replay
```

### Dead-Letter Queue

A delivery that exhausts all its retries is parked in a durable **dead-letter
queue** instead of being silently dropped, so nothing is lost when an endpoint
is down for an extended window. Operators receive a **batched failure-summary
email** (one digest, not one mail per failure).

```bash
# List parked (dead-lettered) deliveries
GET /api/v1/manage/webhooks/dead-letters

# Replay one once its endpoint is healthy again (reuses the idempotency key)
POST /api/v1/manage/webhooks/dead-letters/:id/replay

# Dismiss one you don't intend to re-send
POST /api/v1/manage/webhooks/dead-letters/:id/dismiss
```

<Callout kind="info">
  CE ships this durable reliability floor: idempotency key, failed-deliveries
  view, a persistent dead-letter queue with per-item replay/dismiss, and batched
  failure-summary email. Pro (`pro-webhooks`) adds **bulk** replay and
  configurable auto-retry policies on top.
</Callout>
