## Overview

Content types are the backbone of roadbeat Studio. Each content type defines the structure, validation rules, and form layout for a category of content — such as News articles, Events, Job Offers, or Recipes.

Unlike traditional CMSs that use a single "post" format, roadbeat enforces **precise, schema-driven content types** with rich metadata, enabling powerful search, filtering, and matching.

## Importing from the Schema Registry

The fastest way to add content types is to import them from the central **Schema Registry**, which maintains a curated catalog of 50+ standardized content types.

<Steps>
  <Step title="Open Content Types">
    Navigate to **Content Types** in the sidebar.
  </Step>
  <Step title="Click Import from Registry">
    Click the **Import from Registry** button in the top-right corner. This opens the Registry Browser overlay.
  </Step>
  <Step title="Browse and search">
    Browse by category (e.g., News & Publications, Events & Activities) or use the search bar. Each content type shows its description, category, and version.
  </Step>
  <Step title="Import">
    Click **Import to Studio** on the content type you want. Studio fetches the metadata schema, form schema, and teaser schema, then creates the content type locally.
  </Step>
</Steps>

<Callout kind="info">
  The Schema Registry connection is configured via the `SCHEMA_REGISTRY_URL` environment variable (default: `http://localhost:3200`).
</Callout>

## Content Type Builder

You can also create content types from scratch using the visual **Content Type Builder**.

The builder uses a 3-panel layout:

1. **Field Palette** (left) — drag field types onto the canvas
2. **Canvas** (center) — arrange and organize fields
3. **Field Settings** (right) — configure the selected field

### Field Validation

Each field supports validation rules:

- **Required** — field must have a value
- **Min/Max Length** — for text fields
- **Min/Max Value** — for number fields
- **Pattern** — regex validation for text
- **Unique** — value must be unique across all content of this type

## Schema Structure

Behind the scenes, every content type is stored as three JSON schemas:

```json
{
  "typeId": "news",
  "name": "News Article",
  "version": "1.0.0",
  "schema": { /* JSON Schema for the content data */ },
  "formSchema": { /* Form layout: tabs, sections, field order, UI hints */ },
  "teaserSchema": { /* Fields included in the teaser for Discovery Nodes */ }
}
```

### Form Schema

The form schema controls how the content editor renders. It supports:

- **Tabs** — top-level tab navigation (e.g., Basic Info, Content, Media, SEO)
- **Sections** — collapsible groups within tabs
- **Basic/Extended mode** — fields marked as `extended` are hidden in basic view

```json
{
  "layout": {
    "type": "tabs",
    "tabs": [
      {
        "id": "basic",
        "label": "Basic Info",
        "icon": "file-text",
        "sections": [
          {
            "id": "identification",
            "label": "Identification",
            "fields": ["title", "slug", "description"]
          }
        ]
      }
    ]
  }
}
```

## Managing Content Types

### Editing

Content types can be modified after creation. Changes to the schema affect how existing content is validated — be careful when removing or renaming fields that already have data.

### Deleting

<Callout kind="alert">
  Deleting a content type also removes **all content** of that type. This action cannot be undone.
</Callout>

### API Access

Content types are available via the Management API:

```bash
# List all content types
GET /api/v1/manage/content-types

# Get a specific content type
GET /api/v1/manage/content-types/:typeId
```

And via the Delivery API for frontends:

```bash
# List content types (API key auth)
GET /api/v1/delivery/content-types
```
